## must_fix（修正必須）

### F1: FileTab 型に isDirty フィールドが存在しない

**影響セクション**: 4-4. isDirtyフラグの管理（useFileTabs拡張）

設計方針書では FileTab 型に `isDirty: boolean` を追加し、`SET_DIRTY` アクションを追加すると記載している。しかし現在の `src/hooks/useFileTabs.ts` の FileTab インターフェース（行36-47）には isDirty が存在せず、FileTabsAction 型（行58-67）にも SET_DIRTY が未定義である。

**提案**: 設計方針書に Before/After の型定義を明示し、実装者が変更箇所を正確に把握できるようにする。reducer の OPEN_FILE/RESTORE/SET_CONTENT の各 case で `isDirty: false` を設定する必要があることも具体的に列挙する。

---

### F2: MarkdownEditor に onDirtyChange コールバックが存在しない

**影響セクション**: 4-3. ファイル内容ポーリング / 6. isDirty伝搬フロー

設計方針書の isDirty 伝搬フローでは MarkdownEditor が `onDirtyChange(isDirty: boolean)` を呼び出すと記載しているが、現在の `EditorProps`（`src/types/markdown-editor.ts` 行75-91）にこのプロパティは存在しない。MarkdownEditor 内では `isDirty = content !== originalContent`（行202）として内部計算されているが、外部への伝搬手段がない。

**提案**: EditorProps に `onDirtyChange?: (isDirty: boolean) => void` を追加し、`useEffect(() => { onDirtyChange?.(isDirty); }, [isDirty, onDirtyChange])` パターンで伝搬する実装方針を設計方針書に明記する。MarkdownWithSearch が props を中継する必要があることも記載すべき。

## should_fix（修正推奨）

### F3: ツリーポーリングの fetchDirectory が FileTreeView 内部に閉じている

**影響セクション**: 4-2. ツリーポーリング（WorktreeDetailRefactored管理）

設計方針書のコード例では `fetchDirectory('')` を呼び出しているが、この関数は FileTreeView.tsx 内部の useCallback（行564-584）であり、WorktreeDetailRefactored から直接利用できない。

**提案**: コード例を `fetch('/api/worktrees/${worktreeId}/tree')` の直接呼び出しに修正する。

---

### F4: files API の Content-Length ヘッダの不整合

**影響セクション**: 4-5. サーバー側304応答（files API）

設計方針書では `Content-Length: String(stat.size)` をレスポンスに追加すると記載しているが、テキストファイルのレスポンスは JSON ラップされるため、stat.size とレスポンスサイズは一致しない。また、テキストファイルパス（行257-276）では現在 fs.stat を呼んでいないため、Last-Modified 取得にも追加の stat 呼び出しが必要。

**提案**: Content-Length はNextResponse.json が自動設定するため削除し、Last-Modified のみ追加する方針に修正する。

---

### F5: ポーリングによる SET_CONTENT dispatch と MarkdownEditor の再マウント競合

**影響セクション**: 4-3. ファイル内容ポーリング / 9. テスト設計

ポーリングが 200 OK で SET_CONTENT を dispatch した場合、MarkdownEditor が再マウントされ編集中テキストが失われる可能性がある。MarkdownEditor は filePath prop で内部的にコンテンツを管理しているため直接的な問題は起きない可能性が高いが、この前提が設計方針書に明記されていない。

**提案**: この前提を設計方針書に明記し、テスト設計にMarkdownファイル編集中のポーリング動作テストを追加する。

---

### F6: テスト設計の isActive prop テストと設計方針の矛盾

**影響セクション**: 9. テスト設計

セクション9の既存テスト修正で「FileTreeView.test.tsx: isActive propのデフォルト値（true）確認」と記載されているが、セクション4-2の F2 対応で FileTreeView には isActive prop を追加しない方針に改訂済み。テスト設計とアーキテクチャ設計が矛盾している。

**提案**: セクション9から isActive prop テストを削除し、WorktreeDetailRefactored のツリーポーリング統合テストに置き換える。
