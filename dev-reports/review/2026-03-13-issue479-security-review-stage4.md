## Detailed Findings

### D4-001 [should_fix] バレルファイルのexport *による@internal関数の意図しない公開範囲拡大

**カテゴリ**: API公開
**場所**: セクション4-3、セクション2 バレルファイル戦略

**説明**: auto-yes-manager.tsには@internalアノテーション付きの関数が10個以上存在する（pollAutoYes, captureAndStrip, detectAndResolve, scheduleNextPoll等）。設計方針書のバレルファイル戦略では`export *`パターンが示唆されている（セクション2のdb.tsの例: `export * from './db/worktree-db'`）。auto-yes-manager.tsのバレルファイルでこのパターンを使用した場合、分割後のauto-yes-poller.tsやauto-yes-state.tsからexportされた@internal関数がバレル経由で全て外部に公開されるリスクがある。

現在のauto-yes-manager.tsでは単一ファイル内のexportであり、消費者がどの関数をimportするかは消費者の判断に委ねられている。しかし、バレルファイルでexport *を使用すると、サブモジュールに新たなexportを追加した際に意図せずバレル経由で外部に漏れる可能性がある。

**提案**: バレルファイルでは`export *`ではなく、公開API関数のみを明示的に列挙してre-exportする。@internal関数はバレルファイルからは再公開しない。テストファイルからのimportはサブモジュールへの直接importとする。

```typescript
// auto-yes-manager.ts (barrel) - recommended pattern
export { startAutoYesPolling, stopAutoYesPolling } from './auto-yes-poller';
export { getAutoYesState, setAutoYesState, deleteAutoYesState } from './auto-yes-state';
export type { AutoYesState, AutoYesPollerState } from './auto-yes-state';
// @internal functions are NOT re-exported
```

### D4-002 [should_fix] isValidWorktreeId移動時にWORKTREE_ID_PATTERNも同時移動する設計が未明記

**カテゴリ**: バリデーション
**場所**: セクション4-3 isValidWorktreeIdの移動

**説明**: isValidWorktreeIdは`WORKTREE_ID_PATTERN`（`/^[a-zA-Z0-9_-]+$/`）に依存している。設計方針書ではisValidWorktreeIdのpath-validator.tsへの移動を明記しているが、WORKTREE_ID_PATTERNの移動先については記載がない。

path-validator.tsには既にisPathSafe, validateWorktreePath, resolveAndValidateRealPathといったパスバリデーション関数が集約されている。isValidWorktreeIdとWORKTREE_ID_PATTERNをここに追加することで、セキュリティ関連のバリデーションロジックが一箇所に集約される。

WORKTREE_ID_PATTERNがauto-yes-manager.tsに残ると、path-validator.tsからauto-yes-manager.tsへの逆方向importが発生する可能性があり、依存方向の一貫性が損なわれる。

**提案**: WORKTREE_ID_PATTERNもisValidWorktreeIdと共にpath-validator.tsに移動する方針を設計方針書セクション4-3に明記する。

### D4-003 [nice_to_have] isValidWorktreeId移動のアトミック性に関する注意事項の欠如

**カテゴリ**: バリデーション
**場所**: セクション4-3 importパス変更方針（D1-006対応）

**説明**: isValidWorktreeIdの移動と11のAPIルートのimportパス変更は同一コミットで行う必要がある。中間状態ではコンパイルエラーとなり本番デプロイのリスクは低いが、マージコンフリクト時に不整合が生じる可能性がある。

**提案**: 実装チェックリストに「isValidWorktreeIdの移動とimportパス変更は同一コミットで実施する」旨を追記する。

### D4-005 [nice_to_have] db.ts分割時のSQLインジェクション防御パターンの維持確認が未記載

**カテゴリ**: OWASP
**場所**: セクション12 受け入れ基準、セクション5 セキュリティ設計

**説明**: db.ts分割により、SQLクエリを含む関数が4つのファイルに分散する。現在better-sqlite3のプリペアドステートメントでSQLインジェクションを防御しているが、分割作業中にこのパターンが崩れないことの確認基準が受け入れ基準に含まれていない。

**提案**: 既存テストのパスで実質的にカバーされるため必須ではないが、セキュリティ設計セクションに「プリペアドステートメントパターンの維持確認」を明記することを検討する。
