### F2 [should_fix] 304応答のCache-Controlヘッダ未設定によるキャッシュポイズニングリスク

**OWASP**: A05:2021 Security Misconfiguration
**対象セクション**: セクション4-6 サーバー側304応答（files API）

304応答と200応答のいずれにもCache-Controlヘッダの設定設計が含まれていない。リバースプロキシ経由でアクセスする構成では、プロキシが応答をキャッシュし、異なる認証コンテキストのユーザーに返す可能性がある。

**改善提案**: 304応答と200応答の両方に `Cache-Control: no-store, private` ヘッダを追加する設計を明記する。

```typescript
// 304応答
return new Response(null, {
  status: 304,
  headers: {
    'Last-Modified': lastModified,
    'Cache-Control': 'no-store, private',
  },
});

// 200応答
return NextResponse.json(data, {
  headers: {
    'Last-Modified': lastModified,
    'Cache-Control': 'no-store, private',
  },
});
```

### F7 [should_fix] If-Modified-Sinceヘッダの日時パース安全性の未考慮

**OWASP**: A04:2021 Insecure Design
**対象セクション**: セクション4-6 サーバー側304応答（files API）

不正なIf-Modified-Since値（Invalid Date）が送信された場合の挙動が設計方針書に明記されていない。`new Date(ifModifiedSince)` がInvalid Dateを返すと `stat.mtime <= clientDate` はfalseとなり200応答にフォールバックするため安全だが、明示的なバリデーションを追加すべき。

**改善提案**: If-Modified-Sinceのパース後にisNaNチェックを追加し、Invalid Dateの場合は304判定をスキップする設計を明記する。

```typescript
const ifModifiedSince = request.headers.get('If-Modified-Since');
if (ifModifiedSince) {
  const clientDate = new Date(ifModifiedSince);
  // Invalid Date の場合は304判定をスキップ（200応答にフォールバック）
  if (!isNaN(clientDate.getTime()) && stat.mtime <= clientDate) {
    return new Response(null, {
      status: 304,
      headers: { 'Last-Modified': lastModified, 'Cache-Control': 'no-store, private' },
    });
  }
}
```
